@toony1908/stacks-passkey-wallet

A passkey (WebAuthn PRF)-derived Stacks hot-wallet SDK for web and React Native, sharing one framework-agnostic core. No browser extension, no seed phrase to write down at setup — the wallet's key material is derived on demand from the user's platform passkey, used once, and discarded.

The headline: the same synced passkey (iCloud Keychain / Google Password Manager) plus the same prfSalt derives the same wallet on web and mobile. Connect on your phone, open the same app in a browser, hit “Use an existing passkey,” and you're looking at the same address — no seed phrase, no manual import.

Why

Built for one thing: a wallet that just needs a face

No seed phrases at setup, no extension to install, one shared core between web and mobile.

🔑

Passkey-derived, no seed phrase

The wallet's entropy comes from the user's platform passkey + a PRF salt — nothing to write down at setup, nothing to lose on paper.

⏱️

Per-operation key, never stored

The signing key is derived fresh for every send/sign/reveal via withWalletKey, then the raw entropy is zeroized (entropy.fill(0)) in a finally block.

📱

Web + React Native, one core

@toony1908/stacks-passkey-core has no React, no DOM, no WebAuthn — both platform packages feed it 32 bytes of PRF entropy and it does the rest.

⚙️

Fully configurable

Every identity, endpoint, derivation, behavior, appearance, and storage knob in PasskeyWalletConfig is overridable — see the full Configuration reference.

🎨

Light/dark UI components included

WalletButton/WalletDrawer ship a self-injecting, Xverse-style theme on both platforms — dark by default, with a built-in light palette and 'auto' OS-follow via colorScheme. No CSS file or stylesheet to wire up.

🔀

Testnet/mainnet at runtime

Both addresses are derived from one passkey prompt; flipping setNetwork afterwards needs no new prompt.

Architecture

Three packages, one derivation core

Each platform does its own passkey/PRF dance, then hands 32 bytes of entropy to the same shared logic.

PackageWhat it is
@toony1908/stacks-passkey-coreFramework-agnostic shared primitives — config, validation, BIP39/BIP44 derivation, Hiro balance/transaction fetching. No React, no DOM, no WebAuthn.
@toony1908/stacks-passkey-reactWeb SDK — WebAuthn PRF passkeys, a React provider/hooks, and a DOM WalletButton/WalletDrawer UI.
@toony1908/stacks-passkey-react-nativeReact Native SDK — native passkeys via react-native-passkeys (PRF), the same provider/hooks, and an RN WalletButton/WalletDrawer UI.

     │  WebAuthn PRF                     │  native passkey PRF
     │  (navigator.credentials)          │  (react-native-passkeys)
     ▼                                   ▼
  @toony1908/stacks-passkey-react     @toony1908/stacks-passkey-react-native
     │                                   │
     └────────────┬──────────────────────┘
                   │  same 32 bytes of PRF entropy
                   ▼
       @toony1908/stacks-passkey-core
       BIP39 mnemonic → BIP32 derive (m/44'/5757'/0'/0) → sign

✓ Validated end-to-end on a real Android device — including a real mainnet STX send.

The per-operation key model and entropy zeroization are identical on both platforms: a signing key is never cached across operations. It's derived inside a single withWalletKey(credentialId, config, fn) call — triggering a fresh biometric prompt — handed to fn, and the raw entropy is zeroed the instant fn returns or throws. Only public data ({ credentialId, addresses }) is ever persisted: localStorage on web, AsyncStorage on React Native.

Quickstart

Provider + one component

Wrap your app, drop in WalletButton — it renders a connect button, then a pill that opens the full drawer (balance, send, receive, activity, recovery).

tsx
import { StacksPasskeyProvider, WalletButton } from '@toony1908/stacks-passkey-react';

export function App() {
  return (
    <StacksPasskeyProvider appName="My App" network="testnet">
      <WalletButton />
    </StacksPasskeyProvider>
  );
}

Headless: hooks only

tsx
import { useStacksPasskeyWallet, useStxBalance } from '@toony1908/stacks-passkey-react';

function SendPanel() {
  const { isConnected, connect, address, sendStx } = useStacksPasskeyWallet();
  const { balanceMicroStx, isLoading } = useStxBalance();

  if (!isConnected) {
    return <button onClick={() => connect()}>Connect wallet</button>;
  }

  async function handleSend() {
    // amount is in microSTX (1 STX = 1_000_000n microSTX)
    const txid = await sendStx({ recipient: 'ST2JHG...', amount: 1_000_000n });
    console.log('broadcast:', txid);
  }

  return (
    <div>
      <p>{address}</p>
      <p>{isLoading ? 'Loading…' : `${balanceMicroStx} microSTX`}</p>
      <button onClick={handleSend}>Send 1 STX</button>
    </div>
  );
}

Security

Threat model — read before shipping

This SDK derives a hot wallet from a passkey. It's designed to minimize the key's exposure window, not to eliminate every risk.

🔁 Per-operation derivation

The signing key is never generated once and held. Every send/sign/reveal calls withWalletKey(credentialId, cfg, fn), which derives entropy (triggering a fresh biometric prompt), hands a wallet to fn, and zeroes the entropy (entropy.fill(0)) in a finally block — even if fn throws.

📦 Only public data persisted

Storage holds exactly { credentialId, addresses: { mainnet, testnet } } — never a key, mnemonic, or entropy. Never cached in memory across operations either: not in a module variable, React state, ref, or context.

⚠ Honest JS caveat

Entropy bytes are zeroable — the derived mnemonic/private-key strings are not (JS strings are immutable). An XSS payload executing mid-operation (the split second between prompt approval and finally) could theoretically still read that key. Per-operation derivation shrinks this window to milliseconds; it does not eliminate it.

🛡 Phishing resistance

WebAuthn binds the passkey to the relying-party id (rpId, defaulting to your origin). A different domain — even a convincing phishing clone — cannot invoke your passkey or derive your wallet's key material.

Recommendations for consumers

  • Serve your app over HTTPS in production (required anyway for WebAuthn's secure-context check).
  • Ship a strict Content-Security-Policy — this SDK's own attack surface (XSS reading key material mid-operation) is exactly what a good CSP mitigates.
  • Pin your @stacks/transactions and @stacks/wallet-sdk peer dependency versions. Duplicate copies in your dependency tree can cause subtle serialization/signing bugs.
  • Audit your full dependency tree periodically — any script running in your page can attempt to read data during a live operation.
  • Treat this as a hot wallet — not for large or long-term balances.

React Native

Device-validated specifics

What changed to make native passkeys actually work end-to-end, found by running the flow on a real Android device.

  • Native get() needs an explicit rpId. Unlike the browser (which defaults to the page origin), derivePrfEntropy and reconnectWallet's native assertion calls must pass rpId directly.
  • PRF-at-create with a get() fallback. registerPasskey evaluates PRF during create() (the Android Credential Manager pattern) and returns { credentialId, entropy }, so connect() can skip a follow-up get() when the platform supports it — falling back to derivePrfEntropy's get() otherwise.
  • Hermes-safe derivation. walletFromEntropy derives the STX key directly via @scure/bip39 + @scure/bip32 (m/44'/5757'/0'/0) instead of @stacks/wallet-sdk's generateWallet, which calls encryptMnemonic and needs Web Crypto's crypto.subtle — unavailable in Hermes. Same derived address either way (pinned snapshot test).
  • Dual ESM+CJS build for Metro. Both @toony1908/stacks-passkey-core and @toony1908/stacks-passkey-react-native ship dist/index.js (ESM) and dist/index.cjs (CJS) with a react-native field, so Metro's resolver gets a build it can bundle.
  • AsyncStorage by default. Unlike web's synchronous localStorage, storage here is async (@react-native-async-storage/async-storage) — there's necessarily one brief render where isConnected is false before the mount effect resolves.
Extension Points

STX-only today — everything else via withWalletKey

SIP-10 tokens, NFTs, arbitrary message signing, and contract calls aren't built in, but they're all reachable through withWalletKey, which hands your callback a one-shot { privateKey, address, mnemonic }.

tsx
import { useStacksPasskeyWallet } from '@toony1908/stacks-passkey-react';
import { makeContractCall, broadcastTransaction } from '@stacks/transactions';

function useCallMyContract() {
  const { withWalletKey, network } = useStacksPasskeyWallet();

  return () =>
    withWalletKey(async ({ privateKey }) => {
      const tx = await makeContractCall({
        contractAddress: 'ST1PQHQKV0RJXZFY1DGX8MNSNYVE3VGZJSRTPGZGM',
        contractName: 'my-contract',
        functionName: 'do-thing',
        functionArgs: [],
        senderKey: privateKey,
        network,
      });
      const result = await broadcastTransaction({ transaction: tx, network });
      if ('reason' in result) throw new Error(`Broadcast failed: ${result.reason}`);
      return result.txid;
    });
}

The same withWalletKey is available from useStacksPasskeyWallet() (scoped to the connected wallet) on either platform, or standalone from the /core entry (withWalletKey(credentialId, config, fn)) for non-React usage.
